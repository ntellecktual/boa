    Design Decisions & Tradeoffs

    Every compatibility strategy has a reason — here's the "why" behind schema evolution architecture.

    Why Backward Compatibility as the Default?

    In event-driven systems, consumers deploy independently from producers. When a producer adds a field, old consumers must still read the new schema (backward compatibility). Making backward compatibility the default ensures zero-downtime deployments: producers upgrade first, and consumers catch up on their own schedule. The tradeoff: backward compatibility forbids removing fields or changing types, which limits schema evolution flexibility.

    COMPATIBILITY DEFAULT
    Why Avro Over Protobuf for Schema Evolution?

    Avro schemas are stored separately from data (in the registry), enabling schema evolution without recompiling consumers. Protobuf embeds field numbers in the wire format, which is more efficient but requires recompilation when schemas change. For high-evolution environments (20+ schema changes/quarter), Avro's runtime schema resolution saves significant operational overhead. The tradeoff: Avro's wire format is larger than Protobuf's (no field number compression).

    SERIALIZATION FORMAT
    Why Schema Registry Instead of Embedded Schemas?

    Without a registry, every message carries its schema (bloating payloads) or consumers must know schemas in advance (tight coupling). The registry acts as a single source of truth: producers register schemas, consumers look them up by ID. This decouples producer/consumer deployment cycles. The tradeoff: the registry becomes a single point of failure — if it's down, new schemas can't be registered and unknown schema IDs can't be resolved.

    CENTRALIZED GOVERNANCE
    Why Full Compatibility for Financial Events?

    Financial events (payments, settlements) have regulatory requirements for data retention and replay. Full compatibility (both backward AND forward) ensures that any consumer version can read any producer version — critical when replaying 7-year-old events for audit. The tradeoff: full compatibility is the most restrictive mode — you can only add optional fields with defaults. No field removal, no type changes.

    REGULATORY COMPLIANCE
    Why Semantic Versioning for Schemas?

    SemVer (MAJOR.MINOR.PATCH) communicates the impact of schema changes: PATCH = documentation/metadata only, MINOR = backward-compatible additions, MAJOR = breaking changes requiring consumer updates. This lets operations teams automate deployment decisions: auto-deploy MINOR/PATCH, require manual approval for MAJOR. Without SemVer, every schema change requires human review.

    Confluent Schema Registry Integration (Python) 
    from confluent_kafka.schema_registry import SchemaRegistryClient, Schema
from confluent_kafka.schema_registry.avro import AvroSerializer, AvroDeserializer
import json, logging

logger = logging.getLogger("schema_registry")

class SchemaManager:
    """Manages Avro schema lifecycle against Confluent Schema Registry."""

    def __init__(self, registry_url: str, auth: tuple = None):
        conf = {'url': registry_url}
        if auth:
            conf['basic.auth.user.info'] = f"{auth[0]}:{auth[1]}"
        self.client = SchemaRegistryClient(conf)

    def register_with_compat_check(self, subject: str,
                                    schema_str: str) -> int:
        """Register schema only if it passes compatibility check."""
        schema = Schema(schema_str, 'AVRO')
        # Pre-flight compatibility test before registration
        try:
            is_compat = self.client.test_compatibility(
                subject, schema
            )
            if not is_compat:
                logger.error(f"Schema incompatible with {subject}")
                raise ValueError(
                    f"Schema breaks {subject} compatibility"
                )
        except Exception as e:
            if '40401' not in str(e):  # First schema — no prior version
                raise
        schema_id = self.client.register_schema(subject, schema)
        logger.info(f"Registered {subject} v{schema_id}")
        return schema_id

    def resolve_for_consumer(self, schema_id: int) -> dict:
        """Resolve schema by ID during consumption (cached)."""
        schema = self.client.get_schema(schema_id)
        return json.loads(schema.schema_str)

    def set_compatibility(self, subject: str,
                          level: str = 'BACKWARD') -> None:
        """Set subject-level compatibility (overrides global)."""
        self.client.set_compatibility(subject, level)
        logger.info(f"{subject} compatibility → {level}")
    Schema Compatibility Checker (Python) 
    from typing import List, Dict, Optional
from dataclasses import dataclass

@dataclass
class CompatIssue:
    field: str
    issue_type: str   # 'REMOVED' | 'TYPE_CHANGED' | 'MISSING_DEFAULT'
    message: str
    breaking: bool = True

def check_backward(writer: dict, reader: dict) -> List[CompatIssue]:
    """Backward: can NEW reader read OLD writer data?
    All writer fields must exist in reader OR have defaults."""
    issues = []
    reader_fields = {f['name']: f for f in reader['fields']}
    writer_fields = {f['name']: f for f in writer['fields']}
    # Writer field removed from reader → break
    for name, wf in writer_fields.items():
        if name not in reader_fields:
            issues.append(CompatIssue(name, 'REMOVED',
                f'Field "{name}" in writer but missing from reader'))
        elif reader_fields[name]['type'] != wf['type']:
            # Handle union promotion: ["null", "string"] compat with "string"
            if not _union_compatible(wf['type'], reader_fields[name]['type']):
                issues.append(CompatIssue(name, 'TYPE_CHANGED',
                    f'"{name}": {wf["type"]} → {reader_fields[name]["type"]}'))
    # New reader fields without defaults → break on old data
    for name, rf in reader_fields.items():
        if name not in writer_fields and 'default' not in rf:
            issues.append(CompatIssue(name, 'MISSING_DEFAULT',
                f'New field "{name}" has no default — old data breaks'))
    return issues

def check_forward(writer: dict, reader: dict) -> List[CompatIssue]:
    """Forward: can OLD reader read NEW writer data?
    Symmetric of backward — old consumers must handle new fields."""
    issues = []
    reader_fields = {f['name']: f for f in reader['fields']}
    writer_fields = {f['name']: f for f in writer['fields']}
    for name, wf in writer_fields.items():
        if name not in reader_fields and 'default' not in wf:
            issues.append(CompatIssue(name, 'MISSING_DEFAULT',
                f'New field "{name}" required — old reader can\'t parse'))
        elif name in reader_fields:
            if reader_fields[name]['type'] != wf['type']:
                if not _union_compatible(wf['type'], reader_fields[name]['type']):
                    issues.append(CompatIssue(name, 'TYPE_CHANGED',
                        f'Type change on "{name}" — forward readers fail'))
    return issues

def check_full(writer: dict, reader: dict) -> List[CompatIssue]:
    """Full: backward + forward — most restrictive mode."""
    return check_backward(writer, reader) + check_forward(writer, reader)

def _union_compatible(t1, t2) -> bool:
    """Check if types are union-compatible (Avro union promotion)."""
    if isinstance(t1, list) and not isinstance(t2, list):
        return t2 in t1
    if isinstance(t2, list) and not isinstance(t1, list):
        return t1 in t2
    return False
    Schema Migration Orchestrator (Python) 
    import time, logging
from typing import List
from dataclasses import dataclass, field
from confluent_kafka.admin import AdminClient

logger = logging.getLogger("migration")

@dataclass
class MigrationPlan:
    subject: str
    old_version: int
    new_version: int
    canary_pct: float = 0.05
    error_threshold: float = 0.001  # 0.1% deser error rate
    monitor_window_sec: int = 900     # 15 min observation

class MigrationOrchestrator:
    """Orchestrates safe schema rollout with canary + rollback."""

    def __init__(self, schema_mgr, metrics_client):
        self.schema_mgr = schema_mgr
        self.metrics = metrics_client

    def execute(self, plan: MigrationPlan) -> bool:
        """Run full migration: validate → canary → monitor → promote/rollback."""

        # Step 1: Validate against ALL active consumer versions
        consumer_versions = self.schema_mgr.get_active_versions(
            plan.subject
        )
        for cv in consumer_versions:
            issues = check_full(cv, plan.new_schema)
            if issues:
                logger.error(
                    f"Incompatible with consumer v{cv['version']}: "
                    f"{[i.message for i in issues]}"
                )
                return False

        # Step 2: Deploy canary (5% traffic gets new schema)
        logger.info(f"Canary deploy: {plan.canary_pct*100}% traffic")
        self._set_canary_routing(plan.subject, plan.canary_pct)

        # Step 3: Monitor deserialization error rates
        start = time.time()
        while time.time() - start < plan.monitor_window_sec:
            error_rate = self.metrics.query(
                f'rate(kafka_consumer_deser_errors_total'
                f'{{subject="{plan.subject}"}}[5m])'
            )
            if error_rate > plan.error_threshold:
                logger.error(
                    f"Error rate {error_rate:.4f} > threshold"
                )
                self._rollback(plan)
                return False
            time.sleep(30)

        # Step 4: Promote — route 100% to new schema
        logger.info("Canary passed — promoting to 100%")
        self._set_canary_routing(plan.subject, 1.0)
        return True

    def _rollback(self, plan: MigrationPlan):
        logger.warning(f"Rolling back {plan.subject} to v{plan.old_version}")
        self._set_canary_routing(plan.subject, 0.0)

    def _set_canary_routing(self, subject: str, pct: float):
        """Configure weighted routing in the schema-aware producer."""
        self.metrics.set_gauge(
            f'schema_canary_weight{{subject="{subject}"}}', pct
        )
    Innovation

    Innovation Spotlight

    Novel techniques that go beyond standard schema registry patterns.

    Schema Compatibility Graph

    Instead of checking compatibility only between adjacent versions (v3 vs v4), build a full compatibility graph checking every version pair. This catches "compatibility chains" where v1→v2 is compatible and v2→v3 is compatible, but v1→v3 is NOT compatible — a real problem when consumers skip versions during upgrades. The graph enables automated detection of transitive incompatibility, letting operations teams identify which consumer versions require mandatory upgrades before a new schema can ship.

    Dead Field Detection — By analyzing the last 30 days of Kafka consumer group offsets and schema usage, the system identifies fields that no active consumer reads. These "dead fields" can be safely deprecated in the next MAJOR version, preventing schema bloat that accumulates over years of evolution. The detector correlates field-level access patterns from the Avro deserializer with consumer lag metrics, producing a confidence score for each field's liveness.
