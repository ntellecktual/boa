Pipeline Phases

Six-Phase MDM Architecture

Each phase is an independently deployable Databricks Job Cluster. Unity Catalog enforces row-level security on PII columns at every layer.

Multi-Source Ingestion

Kafka consumers pull member events from 11 upstream systems — CRM, EHR, enrollment, claims, and provider directories — with Avro schema validation at the topic boundary.

  • Apache Kafka + Azure Event Hubs
  • Schema Registry / Avro contracts
  • Dead-letter queue for schema violations
Bronze Layer — Raw Capture

Auto Loader streams raw JSON/Avro events from ADLS Gen2 into Delta tables with full schema evolution. No records are dropped — every byte is immutably stored for replay.

  • Databricks Auto Loader (cloudFiles)
  • Delta Lake append-only mode
  • Change Data Feed enabled
Silver Layer — Cleanse & Link

PySpark jobs apply 47 DQ rules, standardize addresses and phone numbers via USPS CASS and geospatial lookup, then run probabilistic entity matching to link records across sources.

  • PySpark + Delta Live Tables
  • USPS CASS + geospatial enrichment
  • Probabilistic entity resolution (splink)
Gold Layer — Golden Record

Survivorship rules elect the most authoritative value for each attribute across matched records. Confidence scores, source lineage, and timestamps are preserved on every field.

  • Attribute-level survivorship rules
  • Source trust scoring framework
  • Full lineage in Unity Catalog
API Egress & Integration

FastAPI microservices expose golden records via REST. Downstream systems subscribe via Kafka topics or pull via Delta Sharing. C# .NET consumers integrate via the member profile SDK.

  • FastAPI REST + OpenAPI 3.0
  • Kafka publish for real-time consumers
  • C# / .NET SDK for enterprise consumers
AI-Assisted Data Quality

ML models flag anomalous ingestion patterns, predict match confidence, and surface stewardship queues. Automated remediation handles 91% of DQ exceptions without human intervention.

# ============================================================
# Enterprise Member MDM — Delta Live Tables Pipeline
# Medallion: Bronze → Silver → Gold (Golden Record)
# ============================================================
import dlt
from pyspark.sql import functions as F
from pyspark.sql.types import StringType, DoubleType
from pyspark.sql.window import Window

# ── Source trust scores for survivorship ──────────────────
SOURCE_TRUST = {
    "salesforce_crm":   0.92,
    "epic_ehr":         0.97,   # clinical system = highest trust
    "enrollment_svc":   0.95,
    "claims_adjudicate":0.88,
    "provider_dir":     0.80,
    "legacy_mainframe": 0.60,
}

# ── BRONZE: Raw member events from Kafka / ADLS Gen2 ─────
@dlt.table(
    name="bronze_member_events",
    comment="Raw member events from all 11 source systems",
    table_properties={"delta.enableChangeDataFeed": "true"}
)
def bronze_member_events():
    return (
        spark.readStream
        .format("cloudFiles")
        .option("cloudFiles.format", "avro")
        .option("cloudFiles.schemaLocation", "abfss://schemas@mdmstorage.dfs.core.windows.net/member")
        .option("cloudFiles.schemaEvolutionMode", "addNewColumns")
        .load("abfss://raw@mdmstorage.dfs.core.windows.net/member-events/")
        .withColumn("_ingest_ts", F.current_timestamp())
        .withColumn("_source_file", F.input_file_name())
    )

# ── SILVER: Cleansed, validated, geo-enriched ─────────────
@dlt.table(name="silver_members")
@dlt.expect_or_drop("valid_member_id",    "member_id IS NOT NULL")
@dlt.expect_or_drop("valid_dob",         "date_of_birth > '1900-01-01'")
@dlt.expect_or_quarantine("valid_zip",    "LENGTH(zip_code) IN (5, 9)")
@dlt.expect_or_quarantine("valid_phone",  "phone_number RLIKE '^[0-9]{10}$'")
def silver_members():
    # Read from bronze with CDF for incremental processing
    df = dlt.read_stream("bronze_member_events")

    # Standardise & normalise
    df = (df
        .withColumn("first_name", F.initcap(F.trim(F.col("first_name"))))
        .withColumn("last_name",  F.initcap(F.trim(F.col("last_name"))))
        .withColumn("email",       F.lower(F.trim(F.col("email"))))
        .withColumn("zip_code",    F.col("zip_code").substr(1, 5))
        .withColumn("phone_e164",  F.regexp_replace(F.col("phone_number"), "[^0-9]", ""))
    )

    # Geospatial enrichment: zip → lat/lon, county, FIPS
    geo_ref = spark.table("catalog.reference.zip_geospatial")
    df = df.join(F.broadcast(geo_ref), "zip_code", "left")

    # Source trust score for survivorship downstream
    trust_map = F.create_map([F.lit(x) for pair in SOURCE_TRUST.items() for x in pair])
    df = df.withColumn("_source_trust", trust_map[F.col("source_system")])

    return df

# ── GOLD: Golden record via survivorship ──────────────────
@dlt.table(
    name="gold_member_golden_record",
    comment="System of Record — highest-trust attribute per member"
)
def gold_member_golden_record():
    # Window: per enterprise_member_id, ranked by source trust desc
    w = Window.partitionBy("enterprise_member_id").orderBy(F.col("_source_trust").desc())

    return (
        dlt.read("silver_members")
        .withColumn("_rank", F.rank().over(w))
        .filter(F.col("_rank") == 1)        # elect highest-trust record per member
        .withColumn("golden_record_ts", F.current_timestamp())
        .withColumn("record_version", F.expr("uuid()"))
        .drop("_rank", "_ingest_ts", "_source_file")
    )

Design Decisions

Architecture Trade-offs

Key technical decisions made when building an enterprise-scale MDM platform on Azure Databricks.

Streaming vs. Batch Ingestion

Kafka streaming with micro-batches (5-min trigger) instead of nightly batch loads. Member updates in CRM are visible in the golden record within minutes, not hours — critical for real-time care coordination.

Kafka + Auto Loader vs Nightly ADF Batch
Probabilistic vs. Deterministic Matching

Splink-based probabilistic matching using Fellegi-Sunter model trained on 500K labeled pairs. Deterministic rules alone failed on name variations and address reformats common in healthcare data — recall went from 78% to 97%.

Probabilistic (Splink) vs Deterministic Rules
Delta Live Tables vs. Raw Spark Jobs

DLT provides built-in data quality expectations, lineage tracking, and orchestration for the medallion layers. Chose DLT over raw Spark jobs to eliminate boilerplate retry logic and get automatic DQ quarantining without custom code.

Delta Live Tables vs Raw PySpark Jobs
Geospatial Enrichment Strategy

ZIP code resolved to lat/lon, county FIPS, and CBSA code via broadcast-join against a cached USPS CASS reference table (32K rows). Enabled downstream network adequacy analysis and CMS-compliant service area reporting without a GIS microservice round-trip.

Broadcast Join (CASS Ref) vs External GIS API Call
Unity Catalog for PII Governance

Row-level security policies in Unity Catalog restrict PHI columns (SSN, DOB, diagnosis codes) to roles with HIPAA training certification. Column masking applied to non-clinical consumers, eliminating the need for separate de-identified copies.

Unity Catalog RLS / Masking vs Duplicate De-ID Tables
C# SDK for Enterprise Consumers

Published a typed C# NuGet package wrapping the REST golden record API. Downstream .NET billing and claims adjudication systems integrate without raw HTTP calls, reducing integration time from weeks to days and ensuring contract alignment via auto-generated OpenAPI 3.0 spec.

Typed C# SDK (NuGet) vs Raw REST / curl scripts

def ingest_to_adls(batch: list[dict], source: str):
    """Write Avro-decoded events to ADLS Gen2 landing zone."""
    df = spark.createDataFrame(batch)
    (df.write
       .format("delta")
       .mode("append")
       .option("mergeSchema", "true")
       .partitionBy("event_date", "source_system")
       .save(f"abfss://raw@mdmstorage.dfs.core.windows.net/member-events/{source}/"))
Geospatial Enrichment — ZIP to Lat/Lon + FIPS (PySpark) 
from pyspark.sql import functions as F
from pyspark.sql.types import DoubleType, StringType

def enrich_geospatial(df, spark):
    """
    Join member ZIP codes against USPS CASS + Census FIPS reference table.
    Returns enriched DataFrame with lat, lon, county_fips, cbsa_code.
    """
    # Broadcast the small reference table (32K rows, fits in memory)
    geo_ref = (spark.table("catalog.reference.zip_geospatial")
               .select("zip_code", "latitude", "longitude",
                       "county_fips", "cbsa_code", "state_abbr", "county_name"))

    enriched = (df
        .withColumn("zip5", F.col("zip_code").substr(1, 5))
        .join(F.broadcast(geo_ref), df["zip_code"] == geo_ref["zip_code"], "left")
        .withColumn("geo_confidence",
            F.when(F.col("latitude").isNotNull(), F.lit("zip5_centroid"))
             .otherwise(F.lit("unresolved")))
        .drop(geo_ref["zip_code"])
    )
    return enriched

Entity Resolution — Probabilistic Matching (Python / Splink) 
import splink.spark.comparison_library as cl
from splink.spark.spark_linker import SparkLinker

def build_linker(df_spark) -> SparkLinker:
    settings = {
        "link_type": "dedupe_only",
        "comparisons": [
            cl.levenshtein_at_thresholds("last_name",  [1, 2]),
            cl.levenshtein_at_thresholds("first_name", [1]),
            cl.exact_match("date_of_birth"),
            cl.exact_match("zip_code"),
            cl.exact_match("phone_e164"),
        ],
        "blocking_rules_to_generate_predictions": [
            "l.last_name = r.last_name AND l.zip_code = r.zip_code",
            "l.date_of_birth = r.date_of_birth AND l.zip_code = r.zip_code",
        ],
        "retain_matching_columns": True,
        "retain_intermediate_calculation_columns": False,
    }
    linker = SparkLinker(df_spark, settings, spark=spark,
                         catalog="catalog", database="mdm_silver")
    linker.estimate_u_using_random_sampling(max_pairs=1e7)
    return linker

def run_entity_resolution(df) -> DataFrame:
    linker = build_linker(df)
    pairwise = linker.predict(threshold_match_probability=0.85)
    clusters = linker.cluster_pairwise_predictions_at_threshold(pairwise, 0.85)
    return clusters.as_spark_dataframe()
